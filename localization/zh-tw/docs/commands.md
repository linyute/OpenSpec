# 命令 (Commands)

這是 OpenSpec 斜線命令的參考。這些命令可在您的 AI 編碼助理的聊天介面中呼叫 (例如：Claude Code、Cursor、Windsurf)。

有關工作流模式以及何時使用各個命令，請參閱 [工作流 (Workflows)](workflows.md)。有關 CLI 命令，請參閱 [CLI](cli.md)。

## 快速參考

### 預設快速路徑 (`core` 設定檔)

| 命令            | 用途                               |
| --------------- | ---------------------------------- |
| `/opsx:propose` | 在一個步驟中建立變更並生成規劃成品 |
| `/opsx:explore` | 在致力於變更之前先思考想法         |
| `/opsx:apply`   | 實作來自變更的任務                 |
| `/opsx:archive` | 封存已完成的變更                   |

### 擴展工作流命令 (自訂工作流選擇)

| 命令                 | 用途                                      |
| -------------------- | ----------------------------------------- |
| `/opsx:new`          | 開始新的變更鷹架 (scaffold)               |
| `/opsx:continue`     | 根據依賴關係建立下一個成品                |
| `/opsx:ff`           | 快進 (Fast-forward)：一次建立所有規劃成品 |
| `/opsx:verify`       | 驗證實作是否與成品相符                    |
| `/opsx:sync`         | 將增量規格 (delta specs) 合併至主規格     |
| `/opsx:bulk-archive` | 一次封存多個變更                          |
| `/opsx:onboard`      | 引導式教學，帶領您完成整個工作流          |

預設的全域設定檔為 `core`。若要啟用擴展工作流命令，請執行 `openspec config profile`，選取工作流，然後在專案中執行 `openspec update`。

## 命令參考

### `/opsx:propose`

在一個步驟中建立新變更並生成規劃成品。這是 `core` 設定檔中的預設啟動命令。

**語法：**
```text
/opsx:propose [變更名稱或描述]
```

**引數：**
| 引數                         | 必要 | 描述                                |
| ---------------------------- | ---- | ----------------------------------- |
| `change-name-or-description` | 否   | Kebab-case 名稱或一般語言的變更描述 |

**執行內容：**
- 建立 `openspec/changes/<變更名稱>/`
- 生成實作前所需的成品 (對於 `spec-driven`：proposal, specs, design, tasks)
- 在變更準備好進行 `/opsx:apply` 時停止

**範例：**
```text
您：/opsx:propose add-dark-mode

AI：已建立 openspec/changes/add-dark-mode/
     ✓ proposal.md
     ✓ specs/ui/spec.md
     ✓ design.md
     ✓ tasks.md
     準備好進行實作。執行 /opsx:apply。
```

**提示：**
- 使用此命令可獲得最快的一站式路徑
- 如果您想要逐步控制成品，請啟用擴展工作流並使用 `/opsx:new` + `/opsx:continue`

### `/opsx:explore`

在致力於變更之前，先思考想法、調查問題並釐清需求。

**語法：**
```
/opsx:explore [主題]
```

**引數：**
| 引數    | 必要 | 描述                   |
| ------- | ---- | ---------------------- |
| `topic` | 否   | 您想要探索或調查的內容 |

**執行內容：**
- 開啟探索性對話，無需固定結構
- 調查程式碼庫以回答問題
- 比較各個選項和方法
- 建立視覺化圖表以釐清思緒
- 當見解具體化時，可以轉換至 `/opsx:propose` (預設) 或 `/opsx:new` (擴展工作流)

**範例：**
```text
您：/opsx:explore

AI：您想探索什麼？

您：我們應該如何處理行動應用程式的驗證？

AI：讓我調查一下您目前的驗證設定...
     [分析程式碼庫]

     我看到您在網頁應用程式中使用了基於工作階段的驗證。
     對於行動裝置，我們有三個主要選項：

     1. JWT token (無狀態，支援離線)
     2. OAuth2 (如果您需要第三方登入)
     3. 擴展目前的對話 (較簡單，但需要連線能力)

     您的 API 已經設定了 CORS。您對哪個方向感興趣？

您：我們用 JWT 吧。可以為此開始一個變更嗎？

AI：隨時待命。執行 /opsx:propose add-jwt-auth 以開始。
```

**提示：**
- 在需求不明確或需要調查時使用
- 探索期間不會建立任何成品
- 適合在決定之前比較多種方法
- 可以讀取檔案並搜尋程式碼庫

### `/opsx:new`

開始新的變更鷹架。建立變更資料夾，並等待您使用 `/opsx:continue` 或 `/opsx:ff` 生成成品。

此命令屬於擴展工作流命令集 (未包含在預設的 `core` 設定檔中)。

**語法：**
```
/opsx:new [變更名稱] [--schema <結構描述名稱>]
```

**引數：**
| 引數          | 必要 | 描述                                                    |
| ------------- | ---- | ------------------------------------------------------- |
| `change-name` | 否   | 變更資料夾的名稱 (若未提供則提示)                       |
| `--schema`    | 否   | 要使用的工作流結構描述 (預設：來自設定或 `spec-driven`) |

**執行內容：**
- 建立 `openspec/changes/<變更名稱>/` 目錄
- 在變更資料夾中建立 `.openspec.yaml` 元資料 (metadata) 檔案
- 顯示準備好建立的第一個成品模板
- 若未提供則提示輸入變更名稱和結構描述

**建立的內容：**
```
openspec/changes/<變更名稱>/
└── .openspec.yaml    # 變更元資料 (結構描述、建立日期)
```

**範例：**
```
您：/opsx:new add-dark-mode

AI：已建立 openspec/changes/add-dark-mode/
     結構描述：spec-driven

     準備好建立：proposal
     使用 /opsx:continue 建立，或使用 /opsx:ff 一次建立所有成品。
```

**提示：**
- 使用描述性的名稱：`add-feature`, `fix-bug`, `refactor-module`
- 避免使用通用名稱，如 `update`, `changes`, `wip`
- 結構描述也可以在專案設定 (`openspec/config.yaml`) 中設定

### `/opsx:continue`

建立依賴鏈中的下一個成品。一次建立一個成品，以便進行增量進度。

**語法：**
```
/opsx:continue [變更名稱]
```

**引數：**
| 引數          | 必要 | 描述                                  |
| ------------- | ---- | ------------------------------------- |
| `change-name` | 否   | 要繼續哪個變更 (若未提供則從內容推斷) |

**執行內容：**
- 查詢成品依賴圖
- 顯示哪些成品已就緒與受阻
- 建立第一個就緒的成品
- 讀取依賴檔案以獲取內容
- 顯示建立後有哪些內容變為可用

**範例：**
```
您：/opsx:continue

AI：變更：add-dark-mode

     成品狀態：
     ✓ proposal    (已完成)
     ◆ specs       (就緒)
     ◆ design      (就緒)
     ○ tasks       (受阻 - 需要：specs)

     正在建立規格 (specs)...

     [讀取 proposal.md，根據需求建立規格]

     ✓ 已建立 openspec/changes/add-dark-mode/specs/ui/spec.md

     現在可用：tasks
     執行 /opsx:continue 建立下一個成品。
```

**提示：**
- 在進行下一步之前想要檢閱每個成品時使用
- 適合您想要控制的複雜變更
- 多個成品可能同時變為就緒狀態
- 您可以在繼續之前編輯已建立的成品

### `/opsx:ff`

快進成品建立。一次建立所有規劃成品。

**語法：**
```
/opsx:ff [變更名稱]
```

**引數：**
| 引數          | 必要 | 描述                                  |
| ------------- | ---- | ------------------------------------- |
| `change-name` | 否   | 要快進哪個變更 (若未提供則從內容推斷) |

**執行內容：**
- 按依賴順序建立所有成品
- 透過待辦事項清單追蹤進度
- 當所有 `apply-required` (實作必要) 成品完成時停止
- 在建立下一個成品之前讀取每個依賴項

**範例：**
```
您：/opsx:ff add-dark-mode

AI：正在快進 add-dark-mode...

     ✓ 正在建立 proposal.md
     ✓ 正在建立 specs/ui/spec.md
     ✓ 正在建立 design.md
     ✓ 正在建立 tasks.md

     所有規劃成品已完成！
     準備好進行實作。執行 /opsx:apply 以開始。
```

**提示：**
- 在您對要建構的內容有清晰藍圖時使用
- 對於簡單的變更，比 `/opsx:continue` 更快
- 您之後仍可編輯成品
- 適合中小型功能

### `/opsx:apply`

實作來自變更的任務。瀏覽任務清單，撰寫程式碼並勾選項目。

**語法：**
```
/opsx:apply [變更名稱]
```

**引數：**
| 引數          | 必要 | 描述                                  |
| ------------- | ---- | ------------------------------------- |
| `change-name` | 否   | 要實作哪個變更 (若未提供則從內容推斷) |

**執行內容：**
- 讀取 `tasks.md` 並識別未完成的任務
- 逐一處理任務
- 撰寫程式碼、建立檔案、根據需要執行測試
- 以核取方塊 `[x]` 標記任務為已完成

**範例：**
```
您：/opsx:apply

AI：正在實作 add-dark-mode...

     讀取 tasks.md：
     - [ ] 1.1 建立 ThemeContext
     - [ ] 1.2 新增 CSS 自訂屬性
     - [ ] 1.3 實作 localStorage 持久化

     正在處理 1.1：建立 ThemeContext...
     [建立 src/context/ThemeContext.tsx]

     ✓ 1.1 已完成

     正在處理 1.2：新增 CSS 自訂屬性...
     [更新 src/styles/globals.css]

     ✓ 1.2 已完成
     ...
```

**提示：**
- 被中斷後可以從上次離開的地方恢復
- 透過指定變更名稱來進行平行變更
- 完成狀態會在 `tasks.md` 的核取方塊中追蹤

### `/opsx:verify`

驗證實作是否符合您的變更成品。檢查完整性、正確性和連貫性。

**語法：**
```
/opsx:verify [變更名稱]
```

**引數：**
| 引數          | 必要 | 描述                                  |
| ------------- | ---- | ------------------------------------- |
| `change-name` | 否   | 要驗證哪個變更 (若未提供則從內容推斷) |

**執行內容：**
- 從三個維度檢查實作品質
- 在程式碼庫中搜尋實作證據
- 回報類別為 CRITICAL (嚴重)、WARNING (警告) 或 SUGGESTION (建議) 的問題
- 不會阻擋封存，但會顯現問題

**驗證維度：**

| 維度                      | 驗證內容                                             |
| ------------------------- | ---------------------------------------------------- |
| **完整性 (Completeness)** | 所有任務已完成、所有需求已實作、情境已涵蓋           |
| **正確性 (Correctness)**  | 實作符合規格意圖、邊緣案例已處理                     |
| **連貫性 (Coherence)**    | 技術方案 (design) 的決定反映在程式碼中、模式保持一致 |

**範例：**
```
您：/opsx:verify

AI：正在驗證 add-dark-mode...

     完整性
     ✓ tasks.md 中的所有 8 個任務皆已勾選
     ✓ 規格中的所有需求都有對應的程式碼
     ⚠ 情境「系統偏好偵測」沒有測試覆蓋率

     正確性
     ✓ 實作符合規格意圖
     ✓ 情境中的邊緣案例已處理
     ✓ 錯誤狀態符合規格定義

     連貫性
     ✓ 技術方案決定反映在程式碼結構中
     ✓ 命名慣例與 design.md 一致
     ⚠ 技術方案提到「CSS 變數」，但實作使用的是 Tailwind 類別

     摘要
     ─────────────────────────────
     嚴重問題：0
     警告：2
     準備好封存：是 (帶有警告)

     建議：
     1. 為系統偏好偵測新增測試
     2. 更新 design.md 以反映 Tailwind 的使用，或重構為使用 CSS 變數
```

**提示：**
- 在封存之前執行，以儘早發現不符之處
- 警告不會阻擋封存，但會指出潛在問題
- 適合在提交前檢閱 AI 的工作
- 可以揭示成品與實作之間的偏移

### `/opsx:sync`

**選用命令。** 將變更中的增量規格合併至主規格。封存時會根據需要提示同步，因此您通常不需要手動執行。

**語法：**
```
/opsx:sync [變更名稱]
```

**引數：**
| 引數          | 必要 | 描述                                  |
| ------------- | ---- | ------------------------------------- |
| `change-name` | 否   | 要同步哪個變更 (若未提供則從內容推斷) |

**執行內容：**
- 從變更資料夾中讀取增量規格
- 解析 ADDED/MODIFIED/REMOVED/RENAMED 區塊
- 將變更合併至主 `openspec/specs/` 目錄
- 保留增量中未提及的現有內容
- 不會封存變更 (變更仍維持作用中)

**範例：**
```text
您：/opsx:sync

AI：正在同步 add-dark-mode 增量規格...

     正在讀取：openspec/changes/add-dark-mode/specs/ui/spec.md
     目標：openspec/specs/ui/spec.md

     要套用的變更：
     ✓ ADDED: Theme Selection 需求 (2 個情境)
     ✓ ADDED: Accessibility 需求 (1 個情境)

     正在合併...
     ✓ openspec/specs/ui/spec.md 已更新

     變更仍維持作用中。準備好完成時，請執行 /opsx:archive。
```

**何時手動使用：**

| 情境                                   | 是否使用同步？    |
| -------------------------------------- | ----------------- |
| 長期進行的變更，想在封存前讓主規格同步 | 是                |
| 多個平行變更需要更新後的主規格         | 是                |
| 想單獨預覽/檢閱合併結果                | 是                |
| 快速變更，直接進行封存                 | 否 (封存時會處理) |

**提示：**
- 同步是智慧型處理，非複製貼上
- 可以為現有需求新增情境而不產生重複
- 同步後變更仍維持作用中 (未封存)
- 大多數使用者永遠不需要直接呼叫此命令 — 封存時會根據需要顯示提示

### `/opsx:archive`

封存已完成的變更。完成變更並將其移動至封存資料夾。

**語法：**
```
/opsx:archive [變更名稱]
```

**引數：**
| 引數          | 必要 | 描述                                  |
| ------------- | ---- | ------------------------------------- |
| `change-name` | 否   | 要封存哪個變更 (若未提供則從內容推斷) |

**執行內容：**
- 檢查成品完成狀態
- 檢查任務完成情況 (若未完成則發出警告)
- 若尚未同步，則提供同步增量規格的選項
- 將變更資料夾移動至 `openspec/changes/archive/YYYY-MM-DD-<名稱>/`
- 保留所有成品以供稽核追蹤

**範例：**
```text
您：/opsx:archive

AI：正在封存 add-dark-mode...

     成品狀態：
     ✓ proposal.md 存在
     ✓ specs/ 存在
     ✓ design.md 存在
     ✓ tasks.md 存在 (8/8 個任務已完成)

     增量規格：尚未同步
     → 現在同步嗎？(建議使用)

您：是

AI：✓ 規格已同步至 openspec/specs/ui/spec.md
     ✓ 已移動至 openspec/changes/archive/2025-01-24-add-dark-mode/

     變更封存成功。
```

**提示：**
- 封存不會因任務未完成而阻擋，但會發出警告
- 增量規格可以在封存期間或之前進行同步
- 封存的變更會保留下來以供記錄
- 建議先執行 `/opsx:verify` 以發現問題

---

### `/opsx:bulk-archive`

一次封存多個已完成的變更。處理變更之間的規格衝突。

**語法：**
```
/opsx:bulk-archive [變更名稱清單...]
```

**引數：**
| 引數           | 必要 | 描述                                  |
| -------------- | ---- | ------------------------------------- |
| `change-names` | 否   | 要封存的特定變更 (若未提供則提示選取) |

**執行內容：**
- 列出所有已完成的變更
- 在封存前驗證每個變更
- 偵測變更之間的規格衝突
- 透過檢查實際實作內容來解決衝突
- 按時間順序進行封存

**範例：**
```
您：/opsx:bulk-archive

AI：發現 3 個已完成的變更：
     - add-dark-mode (8/8 個任務已完成)
     - fix-login-bug (3/3 個任務已完成)
     - update-footer (2/2 個任務已完成)

     正在檢查規格衝突...
     ⚠ add-dark-mode 和 update-footer 皆變更了 specs/ui/

     正在調查程式碼庫以解決衝突...
     兩項變更皆已實作。將按時間順序合併：
     1. add-dark-mode (建立於 1 月 20 日)
     2. update-footer (建立於 1 月 22 日)

     要封存所有 3 個變更嗎？

您：是

AI：✓ 已封存 add-dark-mode
     ✓ 已封存 fix-login-bug
     ✓ 已封存 update-footer
     規格合併順序：add-dark-mode → update-footer
```

**提示：**
- 適合平行工作流
- 衝突解決具有代理性 (agentic) (會檢查程式碼庫)
- 變更會按建立順序進行封存
- 在覆寫規格內容前會顯示提示

---

### `/opsx:onboard`

引導式上線 (onboarding)，帶領您完成整個 OpenSpec 工作流。使用您實際程式碼庫的互動式教學。

**語法：**
```
/opsx:onboard
```

**執行內容：**
- 透過旁白引導完成一個完整的工作流循環
- 掃描您的程式碼庫以尋找真正的改進機會
- 建立一個帶有真實成品的實際變更
- 實作實際的工作 (小型且安全的變更)
- 封存已完成的變更
- 隨時解釋每個步驟

**階段：**
1. 歡迎與程式碼庫分析
2. 尋找改進機會
3. 建立變更 (`/opsx:new`)
4. 撰寫提案 (proposal)
5. 建立規格 (specs)
6. 撰寫技術方案 (design)
7. 建立任務 (tasks)
8. 實作任務 (`/opsx:apply`)
9. 驗證實作
10. 封存變更
11. 摘要與後續步驟

**範例：**
```
您：/opsx:onboard

AI：歡迎來到 OpenSpec！

     我將引導您使用實際的程式碼庫完成整個工作流。
     我們將尋找一些小地方進行改進，為其建立適當的變更，
     實作它，然後將其封存。

     讓我掃描您的程式碼庫以尋找機會...

     [分析程式碼庫]

     我發現了一些我們可以處理的事項：
     1. 為聯絡表單新增輸入驗證
     2. 改進驗證流程中的錯誤訊息
     3. 為非同步按鈕新增載入狀態

     您對哪一個感興趣？(或建議其他事項)
```

**提示：**
- 最適合學習工作流的新使用者
- 使用真實程式碼，而非玩具範例
- 建立一個您可以保留或捨棄的真實變更
- 完成大約需要 15-30 分鐘

## 不同 AI 工具的命令語法

不同的 AI 工具使用的命令語法略有不同。請使用與您的工具相符的格式：

| 工具          | 語法範例                                                                                        |
| ------------- | ----------------------------------------------------------------------------------------------- |
| Claude Code   | `/opsx:propose`, `/opsx:apply`                                                                  |
| Cursor        | `/opsx-propose`, `/opsx-apply`                                                                  |
| Windsurf      | `/opsx-propose`, `/opsx-apply`                                                                  |
| Copilot (IDE) | `/opsx-propose`, `/opsx-apply`                                                                  |
| Trae          | 以技能為基礎的呼叫，如 `/openspec-propose`, `/openspec-apply-change` (不生成 `opsx-*` 命令檔案) |

各工具的意圖相同，但命令顯示方式可能因整合而異。

> **注意：** GitHub Copilot 命令 (`.github/prompts/*.prompt.md`) 僅在 IDE 擴充功能 (VS Code, JetBrains, Visual Studio) 中可用。GitHub Copilot CLI 目前不直接支援自訂提示檔案 — 詳情及解決方法請參閱 [受支援的工具](supported-tools.md)。

## 舊版命令 (Legacy Commands)

這些命令使用的是舊的「一次性 (all-at-once)」工作流。它們仍然可以使用，但建議使用 OPSX 命令。

| 命令                 | 執行內容                                          |
| -------------------- | ------------------------------------------------- |
| `/openspec:proposal` | 一次建立所有成品 (proposal, specs, design, tasks) |
| `/openspec:apply`    | 實作變更                                          |
| `/openspec:archive`  | 封存變更                                          |

**何時使用舊版命令：**
- 使用舊工作流的現有專案
- 不需要增量建立成品的簡單變更
- 偏好「全有或全無」的方法

**遷移至 OPSX：**
舊版變更可以使用 OPSX 命令繼續。成品結構是相容的。

## 疑難排解

### 「找不到變更 (Change not found)」

命令無法識別要處理哪個變更。

**解決方法：**
- 明確指定變更名稱：`/opsx:apply add-dark-mode`
- 檢查變更資料夾是否存在：`openspec list`
- 確認您位於正確的專案目錄

### 「沒有成品就緒 (No artifacts ready)」

所有成品皆已完成，或因缺少依賴關係而受阻。

**解決方法：**
- 執行 `openspec status --change <名稱>` 以查看受阻原因
- 檢查所需的成品是否存在
- 先建立缺少的依賴成品

### 「找不到結構描述 (Schema not found)」

指定的結構描述不存在。

**解決方法：**
- 列出可用的結構描述：`openspec schemas`
- 檢查結構描述名稱的拼寫
- 如果是自訂結構描述，請先建立：`openspec schema init <名稱>`

### 命令無法辨識

AI 工具不辨識 OpenSpec 命令。

**解決方法：**
- 確保 OpenSpec 已初始化：`openspec init`
- 重新生成技能：`openspec update`
- 檢查 `.claude/skills/` 目錄是否存在 (針對 Claude Code)
- 重啟您的 AI 工具以載入新技能

### 成品生成不正確

AI 建立了不完整或不正確的成品。

**解決方法：**
- 在 `openspec/config.yaml` 中新增專案內容 (context)
- 新增個別成品規則以提供特定指引
- 在變更描述中提供更多細節
- 使用 `/opsx:continue` 而非 `/opsx:ff` 以獲得更多控制權
