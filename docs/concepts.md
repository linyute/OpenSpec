# 概念 (Concepts)

本指南解釋了 OpenSpec 背後的核心概念以及它們如何整合在一起。有關實際用法，請參閱 [入門指南 (Getting Started)](getting-started.md) 和 [工作流 (Workflows)](workflows.md)。

## 理念

OpenSpec 圍繞著四個原則構建：

```
流動而非僵化 (fluid not rigid)         — 沒有階段閘門，處理有意義的事項
疊代而非瀑布 (iterative not waterfall) — 邊做邊學，隨時精煉
簡單而非複雜 (easy not complex)        — 輕量級設定，極簡程序
現有專案優先 (brownfield-first)        — 適用於現有程式碼庫，而不僅僅是新專案
```

### 為什麼這些原則很重要

**流動而非僵化。** 傳統的規格系統將您鎖定在各個階段：首先規劃，然後實作，最後結束。OpenSpec 更加靈活 — 您可以按照對工作有意義的任何順序建立成品。

**疊代而非瀑布。** 需求會變，理解會加深。在開始時看起來不錯的方法，在看到程式碼庫後可能站不住腳。OpenSpec 擁抱這一現實。

**簡單而非複雜。** 某些規格框架需要廣泛的設定、僵化的格式或沉重的程序。OpenSpec 不會妨礙您。在幾秒鐘內完成初始化，立即開始工作，僅在需要時進行自訂。

**現有專案優先。** 大多數軟體工作並非從頭建構 — 而是修改現有系統。OpenSpec 以增量 (delta) 為基礎的方法，讓指定對現有行為的變更變得容易，而不僅僅是描述新系統。

## 整體架構 (The Big Picture)

OpenSpec 將您的工作組織成兩個主要區域：

```
┌─────────────────────────────────────────────────────────────────┐
│                        openspec/                                │
│                                                                 │
│   ┌─────────────────────┐      ┌──────────────────────────────┐ │
│   │   規格 (specs/)      │      │         變更 (changes/)      │ │
│   │                     │      │                              │ │
│   │  真相來源            │◄─────│  提案變更                     │ │
│   │  系統目前如何運作      │ 合併 │  每個變更 = 一個資料夾         │ │
│   │                     │      │  包含成品 + 增量 (deltas)      │ │
│   │                     │      │                              │ │
│   └─────────────────────┘      └──────────────────────────────┘ │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

**規格 (Specs)** 是真相來源 — 它們描述了您的系統目前的行為。

**變更 (Changes)** 是提案的修改 — 它們存放在獨立的資料夾中，直到您準備好合併它們。

這種分離是關鍵。您可以同時處理多個變更而不會產生衝突。您可以在變更影響主規格之前對其進行檢閱。當您封存一個變更時，它的增量會乾淨地合併到真相來源中。

## 規格 (Specs)

規格使用結構化的需求和情境 (scenarios) 來描述系統的行為。

### 結構

```
openspec/specs/
├── auth/
│   └── spec.md           # 驗證行為
├── payments/
│   └── spec.md           # 付款處理
├── notifications/
│   └── spec.md           # 通知系統
└── ui/
    └── spec.md           # UI 行為和主題
```

按網域 (domain) 組織規格 — 這些邏輯分組應對您的系統有意義。常見模式：

- **按功能區域**：`auth/`, `payments/`, `search/`
- **按元件**：`api/`, `frontend/`, `workers/`
- **按界限上下文 (Bounded Context)**：`ordering/`, `fulfillment/`, `inventory/`

### 規格格式

一個規格包含多個需求，而每個需求都有情境：

```markdown
# 驗證規格 (Auth Specification)

## 目的 (Purpose)
應用程式的驗證和工作階段管理。

## 需求 (Requirements)

### 需求：使用者驗證 (User Authentication)
系統應 (SHALL) 在成功登入時發放 JWT token。

#### 情境：憑證有效
- 假設 (GIVEN) 使用者具有有效的憑證
- 當 (WHEN) 使用者提交登入表單
- 那麼 (THEN) 回傳一個 JWT token
- 並且 (AND) 使用者被導向至儀表板

#### 情境：憑證無效
- 假設 (GIVEN) 憑證無效
- 當 (WHEN) 使用者提交登入表單
- 那麼 (THEN) 顯示錯誤訊息
- 並且 (AND) 不發放任何 token

### 需求：工作階段過期 (Session Expiration)
系統必須 (MUST) 在閒置 30 分鐘後讓工作階段過期。

#### 情境：閒置逾時
- 假設 (GIVEN) 一個已通過驗證的工作階段
- 當 (WHEN) 經過 30 分鐘沒有活動
- 那麼 (THEN) 工作階段失效
- 並且 (AND) 使用者必須重新驗證
```

**關鍵元素：**

| 元素                      | 用途                            |
| ------------------------- | ------------------------------- |
| `## 目的 (Purpose)`       | 此規格網域的高階描述            |
| `### 需求 (Requirement):` | 系統必須具備的特定行為          |
| `#### 情境 (Scenario):`   | 需求在實際運作中的具體範例      |
| SHALL/MUST/SHOULD         | RFC 2119 關鍵字，表示需求的強度 |

### 為什麼要這樣組織規格

**需求是「什麼 (what)」** — 它們陳述系統應該做什麼，而不指定實作方式。

**情境是「何時 (when)」** — 它們提供了可以被驗證的具體範例。良好的情境：
- 是可測試的 (您可以為其撰寫自動化測試)
- 涵蓋了正常路徑和邊緣案例
- 使用假設/當/那麼 (Given/When/Then) 或類似的結構化格式

**RFC 2119 關鍵字** (SHALL, MUST, SHOULD, MAY) 傳達了意圖：
- **MUST/SHALL (必須)** — 絕對必要的需求
- **SHOULD (應)** — 建議，但存在例外
- **MAY (可以)** — 選用的

### 規格是什麼 (以及不是什麼)

規格是**行為契約 (behavior contract)**，而非實作計畫。

良好的規格內容：
- 使用者或下游系統依賴的可觀察行為
- 輸入、輸出和錯誤條件
- 外部約束 (安全性、隱私、可靠性、相容性)
- 可以被測試或明確驗證的情境

在規格中應避免：
- 內部的類別/函式名稱
- 函式庫或框架的選擇
- 逐步的實作細節
- 詳細的執行計畫 (這些應屬於 `design.md` 或 `tasks.md`)

快速測試：
- 如果實作可以變更而不改變外部可見的行為，那麼它可能不屬於規格。

### 保持輕量：進階嚴謹性 (Progressive Rigor)

OpenSpec 旨在避免官僚主義。請使用能讓變更被驗證的最輕量級別。

**精簡規格 (Lite spec，預設值)：**
- 簡短的行為優先需求
- 明確的範圍和非目標
- 幾個具體的驗收檢查

**完整規格 (Full spec，適用於較高風險)：**
- 跨團隊或跨儲存庫的變更
- API/契約變更、遷移、安全性/隱私疑慮
- 模糊性可能導致昂貴重構的變更

大多數變更應保持在精簡模式。

### 人類 + 代理程式協作

在許多團隊中，人類負責探索而代理程式負責草擬成品。預期的循環是：

1. 人類提供意圖、內容 (context) 和約束。
2. 代理程式將其轉換為行為優先的需求和情境。
3. 代理程式將實作細節保留在 `design.md` 和 `tasks.md` 中，而非 `spec.md`。
4. 在實作前，透過驗證確認結構和清晰度。

這讓規格對人類來說易於閱讀，對代理程式來說則保持一致。

## 變更 (Changes)

變更是對系統的提案修改，封裝為包含理解和實作所需一切內容的資料夾。

### 變更結構

```
openspec/changes/add-dark-mode/
├── proposal.md           # 為什麼以及要做什麼
├── design.md             # 如何做 (技術方案)
├── tasks.md              # 實作檢查清單
├── .openspec.yaml        # 變更元資料 (metadata) (選用)
└── specs/                # 增量規格 (delta specs)
    └── ui/
        └── spec.md       # ui/spec.md 中正在變更的內容
```

每個變更都是獨立的。它包含：
- **成品 (Artifacts)** — 捕捉意圖、設計和任務的文件
- **增量規格 (Delta specs)** — 說明哪些內容被新增、修改或移除的規格
- **元資料 (Metadata)** — 此特定變更的選用設定

### 為什麼變更採用資料夾形式

將變更封裝為資料夾有幾個好處：

1. **一切都在一起。** 提案、設計、任務和規格都放在同一個地方。不需要在不同位置搜尋。

2. **平行工作。** 可以同時存在多個變更而不會產生衝突。在 `add-dark-mode` 進行的同時，`fix-auth-bug` 也在處理中。

3. **乾淨的歷史記錄。** 封存時，變更會移至 `changes/archive/`，並保留其完整的內容。您可以回顧並了解不僅發生了哪些變更，還有變更的原因。

4. **對檢閱友善。** 變更資料夾易於檢閱 — 開啟它、閱讀提案、檢查設計、檢視規格增量。

## 成品 (Artifacts)

成品是變更中指引工作的各種文件。

### 成品流程 (The Artifact Flow)

```
提案 (proposal) ──────► 規格 (specs) ──────► 設計 (design) ──────► 任務 (tasks) ──────► 實作 (implement)
       │                    │                  │                  │
    為什麼                是什麼              如何做              步驟
    + 範圍                變更內容            技術方案            要採取哪些步驟
```

成品相互建構。每個成品都為下一個提供內容 (context)。

### 成品類型

#### 提案 (`proposal.md`)

提案在高層次捕捉了**意圖**、**範圍**和**方法 (approach)**。

```markdown
# 提案：新增深色模式

## 意圖
使用者要求新增深色模式選項，以減輕夜間使用的眼部疲勞
並與系統偏好相符。

## 範圍
在範圍內：
- 設定中的主題切換
- 系統偏好偵測
- 將偏好持久化於 localStorage 中

在範圍外：
- 自訂顏色主題 (未來的工作)
- 每一頁的主題覆寫

## 方法
使用 CSS 自訂屬性進行佈景，並搭配 React context
進行狀態管理。在首次載入時偵測系統偏好，
並允許手動覆寫。
```

**何時更新提案：**
- 範圍變更 (縮小或擴大)
- 意圖更加明確 (對問題有更好的理解)
- 方法發生根本性轉變

#### 規格 (位於 `specs/` 中的增量規格)

增量規格描述了相對於目前規格**正在變更的內容**。請參閱下方的 [增量規格](#增量規格)。

#### 設計 (`design.md`)

設計捕捉了**技術方案 (Technical Approach)** 和**架構決策**。

````markdown
# 設計：新增深色模式

## 技術方案
主題狀態由 React Context 管理，以避免屬性鑽取 (prop drilling)。
CSS 自訂屬性可實現執行時切換，而無需切換類別。

## 架構決策

### 決策：使用 Context 而非 Redux
主題狀態使用 React Context 是因為：
- 簡單的二元狀態 (亮色/深色)
- 無複雜的狀態轉移
- 避免增加 Redux 相依性

### 決策：使用 CSS 自訂屬性
使用 CSS 變數而非 CSS-in-JS 是因為：
- 可與現有樣式表搭配使用
- 無執行時開銷
- 瀏覽器原生解決方案

## 資料流
```
ThemeProvider (context)
       │
       ▼
ThemeToggle ◄──► localStorage
       │
       ▼
CSS 變數 (套用於 :root)
```

## 檔案變更
- `src/contexts/ThemeContext.tsx` (新)
- `src/components/ThemeToggle.tsx` (新)
- `src/styles/globals.css` (修改)
````

**何時更新設計：**
- 實作顯示該方法無效
- 發現更好的解決方案
- 相依性或約束發生變更

#### 任務 (`tasks.md`)

任務是**實作檢查清單 (Implementation Checklist)** — 帶有核取方塊的具體步驟。

```markdown
# 任務

## 1. 主題基礎設施
- [ ] 1.1 建立具有亮色/深色狀態的 ThemeContext
- [ ] 1.2 新增顏色的 CSS 自訂屬性
- [ ] 1.3 實作 localStorage 持久化
- [ ] 1.4 新增系統偏好偵測

## 2. UI 元件
- [ ] 2.1 建立 ThemeToggle 元件
- [ ] 2.2 在設定頁面新增切換功能
- [ ] 2.3 更新 Header 以包含快速切換功能

## 3. 樣式
- [ ] 3.1 定義深色主題調色盤
- [ ] 3.2 更新元件以使用 CSS 變數
- [ ] 3.3 測試無障礙環境的對比度
```

**任務最佳實踐：**
- 在標題下對相關任務進行分組
- 使用階層式編號 (1.1, 1.2 等)
- 保持任務足夠小，以便在一次工作階段中完成
- 完成任務後將其勾選

## 增量規格 (Delta Specs)

增量規格是讓 OpenSpec 適用於現有專案開發的關鍵概念。它們描述了**正在變更的內容**，而非重申整個規格。

### 格式

```markdown
# 驗證增量 (Delta for Auth)

## 新增需求 (ADDED Requirements)

### 需求：雙重驗證 (Two-Factor Authentication)
系統必須 (MUST) 支援基於 TOTP 的雙重驗證。

#### 情境：2FA 註冊
- 假設 (GIVEN) 使用者未啟用 2FA
- 當 (WHEN) 使用者在設定中啟用 2FA
- 那麼 (THEN) 顯示用於設定驗證器應用程式的 QR code
- 並且 (AND) 使用者在啟用前必須使用驗證碼進行驗證

#### 情境：2FA 登入
- 假設 (GIVEN) 使用者已啟用 2FA
- 當 (WHEN) 使用者提交有效的憑證
- 那麼 (THEN) 出現 OTP 挑戰
- 並且 (AND) 僅在輸入有效的 OTP 後完成登入

## 修改需求 (MODIFIED Requirements)

### 需求：工作階段過期 (Session Expiration)
系統必須 (MUST) 在閒置 15 分鐘後讓工作階段過期。
(先前為：30 分鐘)

#### 情境：閒置逾時
- 假設 (GIVEN) 一個已通過驗證的工作階段
- 當 (WHEN) 經過 15 分鐘沒有活動
- 那麼 (THEN) 工作階段失效

## 移除需求 (REMOVED Requirements)

### 需求：記住我 (Remember Me)
(已棄用，改用 2FA。使用者在每個工作階段應重新驗證。)
```

### 增量區塊

| 區塊          | 意義         | 封存時的動作     |
| ------------- | ------------ | ---------------- |
| `## 新增需求` | 新行為       | 附加至主規格末尾 |
| `## 修改需求` | 變更後的行為 | 取代現有需求     |
| `## 移除需求` | 棄用的行為   | 從主規格中刪除   |

### 為什麼使用增量而非完整規格

**清晰度。** 增量精確地顯示了正在變更的內容。閱讀完整規格時，您必須在腦中將其與目前版本進行比對。

**避免衝突。** 只要兩個變更修改的是不同的需求，它們就可以同時觸及同一個規格檔案而不會產生衝突。

**檢閱效率。** 檢閱者看到的是變更，而非未變更的內容。這讓焦點能集中在重要事項上。

**適合現有專案。** 大多數工作都是修改現有行為。增量讓修改成為「一等公民」，而非事後才想到的補救。

## 結構描述 (Schemas)

結構描述定義了工作流的成品類型及其依賴關係。

### 結構描述如何運作

```yaml
# openspec/schemas/spec-driven/schema.yaml
name: spec-driven
artifacts:
  - id: proposal
    generates: proposal.md
    requires: []              # 無依賴關係，可優先建立

  - id: specs
    generates: specs/**/*.md
    requires: [proposal]      # 建立前需要有提案

  - id: design
    generates: design.md
    requires: [proposal]      # 可與規格平行建立

  - id: tasks
    generates: tasks.md
    requires: [specs, design] # 需要規格與設計皆完成
```

**成品形成一個依賴圖：**

```
                  提案 (proposal)
                   (根節點)
                       │
         ┌─────────────┴─────────────┐
         │                           │
         ▼                           ▼
    規格 (specs)                設計 (design)
    (需要：                      (需要：
    提案)                       提案)
         │                           │
         └─────────────┬─────────────┘
                       │
                       ▼
                    任務 (tasks)
                   (需要：
                   規格, 設計)
```

**依賴關係是促成因素 (enablers)，而非限制。** 它們顯示了哪些成品是可以建立的，而不是您下一步「必須」建立什麼。如果您不需要設計，可以跳過它。您可以在設計之前或之後建立規格 — 兩者僅依賴於提案。

### 內建結構描述

**規格驅動 (spec-driven)** (預設)

規格驅動開發的標準工作流：

```
提案 → 規格 → 設計 → 任務 → 實作
```

最適合：大多數需要先就規格達成共識再進行實作的功能開發。

### 自訂結構描述

為您團隊的工作流建立自訂結構描述：

```bash
# 從頭建立
openspec schema init research-first

# 或衍生 (fork) 現有的結構描述
openspec schema fork spec-driven research-first
```

**自訂結構描述範例：**

```yaml
# openspec/schemas/research-first/schema.yaml
name: research-first
artifacts:
  - id: research
    generates: research.md
    requires: []           # 先進行研究

  - id: proposal
    generates: proposal.md
    requires: [research]   # 根據研究內容撰寫提案

  - id: tasks
    generates: tasks.md
    requires: [proposal]   # 跳過規格/設計，直接進入任務
```

有關建立和使用自訂結構描述的完整詳細資訊，請參閱 [自訂 (Customization)](customization.md)。

## 封存 (Archive)

封存藉由將增量規格合併到主規格中，並保留變更歷史記錄來完成一項變更。

### 封存時會發生什麼

```
封存前：

openspec/
├── specs/
│   └── auth/
│       └── spec.md ◄────────────────┐
```

### 封存程序

1. **合併增量。** 每個增量規格區塊 (新增/修改/移除) 都會套用到對應的主規格中。

2. **移至封存。** 變更資料夾會移至 `changes/archive/`，並加上日期前綴以便按時間排序。

3. **保留內容。** 所有成品在封存中保持不變。您可以隨時回顧，以了解為什麼要進行某項變更。

### 為什麼封存很重要

**整潔的狀態。** 「作用中變更 (`changes/`)」僅顯示進行中的工作。已完成的工作會被移開。

**稽核追蹤。** 封存保留了每項變更的完整內容 — 不僅僅是變更了什麼，還有解釋為什麼的提案、解釋如何做的設計，以及顯示已完成工作的任務。

**規格演進。** 隨著變更被封存，規格會有機地成長。每次封存都會合併其增量，隨著時間推移建立起一份全面的規格說明。

**良性循環：**

1. 規格描述目前的行為
2. 變更提案修改 (以增量形式)
3. 實作讓變更成真
4. 封存將增量合併至規格中
5. 規格現在描述了新的行為
6. 下一個變更建立在更新後的規格之上

## 詞彙表 (Glossary)

| 術語                           | 定義                                              |
| ------------------------------ | ------------------------------------------------- |
| **成品 (Artifact)**            | 變更中的一份文件 (提案、設計、任務或增量規格)     |
| **封存 (Archive)**             | 完成一項變更並將其增量合併至主規格的程序          |
| **變更 (Change)**              | 對系統的提案修改，封裝為包含成品的資料夾          |
| **增量規格 (Delta spec)**      | 描述相對於目前規格的變更 (新增/修改/移除) 的規格  |
| **網域 (Domain)**              | 規格的邏輯分組 (例如：`auth/`, `payments/`)       |
| **需求 (Requirement)**         | 系統必須具備的特定行為                            |
| **情境 (Scenario)**            | 需求的一個具體範例，通常採用 Given/When/Then 格式 |
| **結構描述 (Schema)**          | 成品類型及其依賴關係的定義                        |
| **規格 (Spec)**                | 描述系統行為的規格說明，包含需求和情境            |
| **真相來源 (Source of truth)** | `openspec/specs/` 目錄，包含目前達成共識的行為    |
